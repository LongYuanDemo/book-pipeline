# 产品思想：项目看板即产物

## 核心洞察

传统项目管理依赖外部工具——Linear、Jira、飞书、Notion。这造成了一个根本矛盾：

**项目状态在代码里，但状态视图在别处。**

工程师改了代码，还要去另一个系统手动更新状态。Agent 执行了任务，无法同步到外部看板。状态和代码之间永远有时差。

## 解决方案

**看板是项目的产物，不是项目的附件。**

看板和代码一起版本控制、一起演进。状态的增删改查由自适应面板维护，不需要外部系统。

## 设计原则

### 1. 状态即代码

项目状态存储在版本控制的文件中（如 `pipeline-status.md`），而非外部数据库。
- Agent 修改了代码 → 状态文件自动更新
- 工程师 review 了模块 → 状态文件手动标注
- Git diff 就能看到状态变更历史

### 2. 自适应而非自定义

看板自动适应项目结构，而非要求项目适配看板。
- 新增模块 → 看板自动添加行列
- 模块下线 → 看板自动移除
- 模块输出格式变化 → 状态计算逻辑自动调整

### 3. 双向同步

一个工程师 + 一群 Agent 通过同一面板同步：
- Agent 执行完 skill → 更新状态文件 → 看板刷新
- 工程师打开看板 → 看到最新状态 → 决定下一步
- 工程师手动标注 review → Agent 下次执行时读取标注

### 4. CRUD 完整

看板不是只读展示，而是完整的状态管理：
- **增**：新模块上线，看板自动识别
- **删**：模块下线，看板自动移除
- **改**：重新执行模块，状态自动更新
- **查**：打开看板即查，无需登录第三方

### 5. 可操作性

看板的每一条状态都指向行动：
- 缺口项附带可复制的修复命令
- 阻塞链显示根因在哪
- 按紧急度排序而非字母序

## 与传统工具的对比

| 维度 | Linear/飞书/Jira | 项目看板 |
|------|-------------------|----------|
| 状态来源 | 人工录入 | 代码自动检测 |
| 同步方式 | 手动更新 | Git 提交即更新 |
| Agent 可访问 | 需 API 集成 | 原生文件读写 |
| 版本历史 | 工具内独立维护 | Git blame |
| 离线可用 | 否 | 是 |
| 上下文切换 | 需切换到外部工具 | 在项目内查看 |
| 自适应 | 固定字段 | 扫描项目结构 |

## 适用场景

- **Pipeline 类项目**：多阶段处理流水线，每阶段有明确输入输出
- **多模块产品**：多个独立功能模块，各自有开发进度
- **Agent 协作项目**：人类 + AI Agent 共同推进，需统一状态视图
- **长期维护项目**：需要追踪历史执行记录和 review 状态

## 不适用场景

- 纯前端展示项目（无工程模块）
- 一次性脚本项目（无持续维护需求）
- 团队规模 > 20 人的项目（需要权限管理和协作流，此时应配合专业工具）
